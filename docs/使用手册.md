# WeCom Codex Bridge Python 使用手册

本文档基于当前仓库实现整理，适用于这个纯 Python 的企业微信智能机器人桥接服务。

项目目录：

```text
/home/jenkins/wecom-bridge/wecom-workspace-bridge-py
```

默认服务地址：

```text
http://127.0.0.1:9299
```

说明：

- 这是一个无前端的 headless 服务，`/` 根路径返回 JSON，不提供网页 UI。
- Bridge 负责把企业微信会话桥接给 `codex`，并补充 Bot 管理、文件回传、会话控制、定时投递和 JSON API。
- 当前实现强调本地部署、单机可用，也支持多个实例共享同一状态目录。

## 1. 功能概览

当前版本支持：

- 企业微信 WebSocket 长连
- 多 Bot 配置加载和持久化
- `codex exec`，以及基于当前运行态的 `codex exec resume` 尝试
- 会话持久化；`threadId` 仅作为运行态恢复线索，不作为稳定 session 身份
- 群聊按成员隔离会话或群共享会话
- 会话控制命令：
  - `/bridge-status`
  - `/bridge-interrupt`
  - `/bridge-reset`
- 图片和文件下载落盘
- 本地命令或 API 触发文件回传
- 一次性定时消息
- cron 周期调度
- Bot / Session / Schedule JSON API
- 本地健康检查、smoke 检查和回归测试

不包含：

- 前端页面
- OAuth 登录流程
- 明文 `secret` 持久化
- `intervalSeconds` 风格的旧周期调度

## 2. 整体架构

Bridge 的核心职责有四类：

1. 与企业微信 Bot 长连，接收用户消息和附件。
2. 为每个会话管理稳定 `sessionId`、运行租约、队列，以及临时 `threadId` 恢复线索。
3. 以当前 Bridge 运行用户身份启动 `codex` 子进程。
4. 提供本地文件发送、定时任务和 Bot 管理 API。

### 2.1 Codex 启动方式

当前实现会执行：

- 新会话：`codex exec --skip-git-repo-check --json -o <output>`
- 恢复会话：如果当前运行态仍保留可用 thread，则尝试 `codex exec resume --skip-git-repo-check --json -o <output>`

执行模式由 `CODEX_EXEC_MODE` 控制：

- `sandboxed`
  说明：Bridge 会加 `--full-auto`
- `host`
  说明：Bridge 会加 `--dangerously-bypass-approvals-and-sandbox`

默认值是 `host`。

### 2.2 Session 工作目录

每个会话运行时会派生出几类目录：

- `workspace/<bot>/sessions/<chat-key-slug>/chatfile`
  说明：当前会话的文件交换区，企微下载的附件和待回传文件都适合放这里
- `workspace/<bot>/users/<user>/workfile`
  说明：用户长期私有工作区
- `workspace/<bot>/rooms/<room>/roomfile`
  说明：群聊级共享工作区
- `PROJECT_DIR/.codex/skills`
  说明：项目级 Codex skills 目录

Bridge 运行 `codex` 时：

- `cwd = WORKFILE_DIR`，纯群共享会话时为 `ROOMFILE_DIR`
- 使用 bridge 管理的 `CODEX_HOME` overlay，但登录态仍继承 bridge 运行用户
- 会把 `TMPDIR` / `TMP` / `TEMP` 指向当前会话的 `chatfile`

### 2.3 关键运行目录

仓库根目录下的重要状态目录：

- `.bots.json`
  说明：Bot 持久化配置
- `.bot-runtime-locks/`
  说明：Bot 运行锁，多实例避免重复占用同一 WeCom Bot
- `.bot-tombstones/`
  说明：跨实例删除事件
- `.session-registry/`
  说明：Session 持久化注册表
- `.session-locks/`
  说明：Session 租约锁
- `.scheduled-messages/`
  说明：定时消息和 schedule definition
- `chatfile/`
  说明：历史兼容目录；当前版本实际会话文件交换目录以 `workspace/.../chatfile` 为准
- `workspace/`
  说明：Bot 级工作目录根
- `bridge.log`
  说明：当前日志
- `bridge.log.prev`
  说明：上一次重启前的日志

## 3. 前置条件

启动前至少确认：

- 已安装 `python3`
- 已安装项目依赖
- 已安装并可执行 `codex`
- Bridge 运行用户已经执行过 `codex login`
- 已拿到企业微信智能机器人的 `botId`
- 已准备好 `secretFile`

推荐检查：

```bash
which python3
which codex
codex login status
echo "${CODEX_HOME:-$HOME/.codex}"
```

安装依赖：

```bash
pip install -r requirements.txt
```

测试依赖：

```bash
pip install -r requirements-dev.txt
```

### 3.1 部署环境要求

建议部署环境满足：

- 操作系统：Linux，或具备等价 Linux 用户态能力的容器环境
- Python：可直接执行 `python3`
- Codex CLI：可直接执行 `codex`
- Codex 登录态：bridge 运行用户已经执行过 `codex login`
- 网络：部署机需要能访问 `wss://openws.work.weixin.qq.com`
- 端口：`BRIDGE_BIND` 指定的监听地址可成功绑定，默认 `127.0.0.1:9299`
- 文件系统：支持普通文件原子替换 `rename/replace`
- 文件锁：支持 `fcntl.flock`，因为 bot 配置锁、session 锁、schedule definition 锁都依赖它
- 时区数据：如果 cron 调度使用 `Asia/Shanghai` 等非 `UTC` 时区，系统需要可用的 IANA timezone 数据；精简镜像通常需要安装 `tzdata`

目录和持久化要求：

- 项目目录需要可写；bridge 会写入 `.bots.json`、`.session-registry/`、`.session-locks/`、`.scheduled-messages/`、`workspace/`、`bridge.log`
- `WECOM_BOT_SECRET_FILE` 指向的 secret 文件必须存在且对 bridge 运行用户可读
- `workDir` 必须存在；它作为共享项目根使用。如果要接收下载文件、发送本地文件、使用会话 workspace，通常应确保 bridge 运行用户对相关目录有写权限
- 如果使用全局 skills，请放到 `~/.codex/skills/`
- 如果使用当前会话/工作区私有 skills，请放到 `PROJECT_DIR/.codex/skills`
- 如果使用容器部署且希望配置、session、schedule、日志在重启后保留，应把上述状态目录挂载到持久卷

代理要求：

- 如需代理出网，在启动前设置 `HTTP_PROXY`、`HTTPS_PROXY`、`NO_PROXY`
- 若 bridge 仅监听本机，`NO_PROXY` 建议包含 `127.0.0.1` 和 `localhost`

## 5. 配置说明

### 5.1 必备配置

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `BRIDGE_BIND` | `127.0.0.1:9299` | 推荐使用的统一监听配置，格式 `host:port`，IPv6 使用 `[host]:port` |
| `BRIDGE_HOST` | 空 | 可选，作为 namespaced 监听地址，优先级低于 `BRIDGE_BIND` |
| `BRIDGE_PORT` | 空 | 可选，作为 namespaced 监听端口，优先级低于 `BRIDGE_BIND` |
| `HOST` | `127.0.0.1` | HTTP 监听地址 |
| `PORT` | `9299` | HTTP 监听端口 |
| `WORK_DIR` | `/home/jenkins` | 默认 `workDir`，表示共享项目根 |
| `CODEX_EXEC_MODE` | `host` | `sandboxed` 或 `host` |
| `WECOM_BOT_ID` | 空 | 单 Bot 启动注入用 |
| `WECOM_BOT_SECRET_FILE` | 空 | 单 Bot secret 文件路径 |

说明：

- 推荐只配置 `BRIDGE_BIND`，这样切换端口时只改一个地方。
- `BRIDGE_BIND` 优先级最高；未配置时依次回退到 `BRIDGE_HOST`/`BRIDGE_PORT`、`HOST`/`PORT`、内置默认值。

### 5.2 API 鉴权

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `BRIDGE_TOKEN` | 空 | Bearer Token 鉴权 |
| `BRIDGE_BASIC_AUTH` | 空 | Basic Auth，格式 `username:password` |

规则：

- 如果 `BRIDGE_TOKEN` 或 `BRIDGE_BASIC_AUTH` 任一已配置，所有 `/api/*` 请求都必须鉴权。
- 如果都未配置，则 API 只允许 localhost 来源访问。
- `BRIDGE_BASIC_AUTH` 和 `BRIDGE_TOKEN` 可以同时启用。

### 5.3 Bot 持久化和引导加载

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `BOTS_FILE` | `./.bots.json` | Bot 持久化文件位置 |
| `WECOM_BOT_CONFIG_ID` | 自动生成 | 单 Bot 的配置 ID；不填时按 `botId` 生成稳定 ID |
| `WECOM_BOT_NAME` | `default` | 单 Bot 名称 |
| `WECOM_BOT_WORK_DIR` | `WORK_DIR` | 单 Bot 的工作目录 |
| `WECOM_BOT_WELCOME` | 空 | 会话欢迎语 |
| `WECOM_BOT_GROUP_SESSION_MODE` | `per-user` | `per-user` 或 `shared` |
| `WECOM_BOT_ENABLED` | `true` | 单 Bot 启用状态 |
| `WECOM_BOOTSTRAP_BOTS_JSON` | 空 | 直接传多 Bot JSON 数组 |
| `WECOM_BOOTSTRAP_BOTS_JSON_FILE` | 空 | 从文件读取多 Bot JSON 数组 |

注意：

- `WECOM_BOT_SECRET` 已不再支持。
- `WECOM_BOOTSTRAP_BOTS_JSON` 和 `WECOM_BOOTSTRAP_BOTS_JSON_FILE` 不能同时设置。
- `WECOM_BOT_ID` 和 `WECOM_BOT_SECRET_FILE` 必须一起设置。
- `.bots.json` 不再写入明文 `secret`。
- 旧版 `.bots.json` 如果残留 `secret`，当前版本会拒绝加载。
- 推荐始终使用 `secretFile`，这样服务重启后可恢复。

### 5.4 文件与消息大小限制

| 变量 | 默认值 | 说明 |
| --- | --- | --- |
| `MAX_JSON_BODY` | `1048576` | API JSON 请求体最大字节数 |
| `MAX_UPLOAD_SIZE` | `104857600` | 文件回传上限，单位字节 |
| `MAX_INBOUND_IMAGE_SIZE` | `31457280` | 企业微信下载图片大小上限 |
| `MAX_INBOUND_FILE_SIZE` | `104857600` | 企业微信下载普通文件大小上限 |
| `FILE_SEND_ROOTS` | 空 | 额外允许回传的目录，逗号分隔 |

说明：

- `workDir` 本身不会自动加入文件回传白名单。
- 默认允许回传的目录只有当前会话的 `chatfile/`。
- 如果需要从其他目录回传文件，必须显式设置 `FILE_SEND_ROOTS=/path/a,/path/b`。

## 6. Bot 配置与生命周期

### 6.1 Bot 配置字段

新增或引导一个 Bot 时，主要字段有：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 否 | Bridge 内部 Bot 配置 ID；通常可不传 |
| `name` | 否 | Bot 名称，默认 `unnamed` 或环境变量中的 `default` |
| `botId` | 是 | 企业微信 Bot ID |
| `secretFile` | 是 | secret 文件路径 |
| `workDir` | 否 | 该 Bot 的工作目录 |
| `welcome` | 否 | 会话欢迎语 |
| `groupSessionMode` | 否 | `per-user` 或 `shared` |
| `enabled` | 否 | 是否启用 |

校验规则：

- `workDir` 必须存在，且必须是目录
- `secretFile` 必须存在、可读、非空
- `groupSessionMode` 只能是 `per-user` 或 `shared`

### 6.2 推荐的多 Bot 引导文件

`WECOM_BOOTSTRAP_BOTS_JSON_FILE` 示例：

```json
[
  {
    "id": "bot-a",
    "name": "bot-a",
    "botId": "BOT_A",
    "secretFile": "/run/secrets/bot_a.secret",
    "workDir": "/home/jenkins",
    "groupSessionMode": "per-user",
    "enabled": true
  },
  {
    "id": "bot-b",
    "name": "bot-b",
    "botId": "BOT_B",
    "secretFile": "/run/secrets/bot_b.secret",
    "workDir": "/home/jenkins",
    "groupSessionMode": "shared",
    "enabled": true
  }
]
```

### 6.3 Bot 状态说明

`GET /api/bots` 里常见的 `status`：

- `running`
  说明：当前实例已连上企业微信并持有运行锁
- `connecting`
  说明：正在连接或重连
- `standby`
  说明：配置存在，但运行锁被其他实例持有
- `disconnected`
  说明：当前实例持有运行锁，但连接已断开
- `error`
  说明：连接、订阅或运行期间发生错误
- `stopped`
  说明：当前实例已停止该 Bot
- `not_loaded`
  说明：Bot 配置在持久化文件中，但当前实例尚未装载
- `disabled`
  说明：Bot 配置存在，但被禁用

### 6.4 多实例行为

如果多个 Bridge 共享同一个状态目录：

- 同一个 `botId` 同时只会有一个实例占用运行锁
- 没拿到运行锁的实例显示为 `standby`
- `stop`、配置更新和删除事件会传播到其他实例
- `delete` 会先发全局删除事件，再清理 Bot、Session 和 Schedule 状态

## 7. Session 模型

### 7.1 `chatKey` 规则

Bridge 对外推荐优先使用 `chatKey`，`sessionId` 作为兜底。

`groupSessionMode=per-user` 时：

- 单聊：`single:USER_ID`
- 群聊：`group-user:CHAT_ID:USER_ID`

`groupSessionMode=shared` 时：

- 群聊：`group:CHAT_ID`

含义：

- `per-user`
  说明：群里不同成员 `@bot` 各自独立会话，但回复仍回原群；主动回传的 markdown 消息会自动 `@` 对应触发成员
- `shared`
  说明：整个群共享一个会话

### 7.2 `sessionId` 和 `threadId`

- `sessionId`
  说明：Bridge 内部会话标识，适合持久化和 API 兜底
- `threadId`
  说明：对应当前运行态里的 Codex thread，仅作为临时恢复线索；不应视为稳定 session 身份的一部分

### 7.3 会话控制命令

以下文本命令会被 Bridge 拦截，不会转发给 Codex：

- `/bridge-status`
- `/bridge-interrupt`
- `/bridge-reset`

行为说明：

- `/bridge-status`
  说明：返回当前会话状态、队列长度、定时任务数、`sessionId`、`threadId`
- `/bridge-interrupt`
  说明：中断当前运行任务，但保留会话上下文和聊天记录
- `/bridge-reset`
  说明：中断当前任务，清除 `threadId`、聊天记录、队列和待处理附件上下文

## 8. API 使用说明

### 8.1 鉴权方式

Basic Auth：

```bash
curl -s http://127.0.0.1:9299/api/bots \
  -u 'bridge:change-me'
```

Bearer Token：

```bash
curl -s http://127.0.0.1:9299/api/bots \
  -H 'Authorization: Bearer YOUR_BRIDGE_TOKEN'
```

### 8.2 根路径

请求：

```bash
curl -s http://127.0.0.1:9299/
```

用途：

- 判断服务存活
- 查看运行模式和并发上限

### 8.3 Bot 管理

#### 获取 Bot 列表

```bash
curl -s http://127.0.0.1:9299/api/bots \
  -u 'bridge:change-me'
```

返回项包含：

- `id`
- `name`
- `botId`
- `workDir`
- `groupSessionMode`
- `secretSource`
- `status`
- `enabled`

### 8.5 文件回传 API

#### 发送文件（主 `bridge.py` 服务）

```bash
curl -s -X POST http://127.0.0.1:9299/api/send-file \
  -u 'bridge:change-me' \
  -H 'Content-Type: application/json' \
  -d '{
    "chatKey": "group-user:CHAT_ID:USER_ID",
    "filePath": "/absolute/path/to/report.md"
  }'
```

也支持 `sessionId` 兜底：

```json
{
  "sessionId": "SESSION_ID",
  "filePath": "/absolute/path/to/report.md"
}
```

可选字段：

- `botName`
  说明：当同一个 `chatKey` 可能匹配多个 Bot 时建议传入

返回示例：

```json
{
  "ok": true,
  "message": "queued report.md",
  "queueDepth": 1
}
```

校验规则：

- `filePath` 必须存在且是普通文件
- 文件大小不能超过 `MAX_UPLOAD_SIZE`
- 文件必须位于允许回传的目录下
- Bot 当前必须处于已连接状态

常见返回：

- `400`：请求体不是合法 JSON，或 `chatKey` / `sessionId` / `filePath` 等前置字段不合法
- `403`：文件不在允许回传目录下
- `404`：文件不存在，或找不到匹配的 Bot / Session
- `409`：`chatKey` 匹配多个 Bot，或目标 Bot / Session 不一致
- `413`：文件超过 `MAX_UPLOAD_SIZE`
- `503`：Bot 未连接，或目标 Bot 当前未运行

说明：

- 上面这组语义对应主 `bridge.py` 服务的 `/api/send-file`
- 主服务采用本地队列模型，请求成功通常表示“已入队并等待 Bridge 处理”
- 因此返回值可能是 `queued ...` 和 `queueDepth`

#### 发送文件（`workspace_bridge.service` 轻量服务）

`workspace_bridge.service` 中同名的 `/api/send-file` 不是队列接口，而是即时发送接口：

- 请求成功表示文件已经完成当前次上传并发送
- 返回值会包含 `sessionId`、`chatKey`、`workspaceId`、`mediaId`
- 常见错误返回是 `400 / 403 / 404 / 413 / 502 / 503 / 504`
- 返回示例通常类似：

```json
{
  "ok": true,
  "sessionId": "SESSION_ID",
  "chatKey": "group-user:CHAT_ID:USER_ID",
  "fileName": "report.md",
  "workspaceId": "WORKSPACE_ID",
  "mediaId": "MEDIA_ID",
  "message": "sent report.md"
}
```

它不支持主 `bridge.py` 那套 `sessionId` 兜底和 `queued` / `queueDepth` 语义，调用时应显式传 `chatKey` 与 `filePath`。

- `runAt`
- `startAt`
- `endAt`

- 结尾 `Z` 可识别为 UTC
- 如果不带时区信息，会按 UTC 解释

### 8.9 常见错误码

| 状态码 | 典型场景 |
| --- | --- |
| `400` | 字段缺失、格式错误、非法 cron、非法 timezone |

| `403` | 非 localhost 且未配置鉴权，或文件不在允许回传目录 |

## 10. 文件和附件处理

### 10.1 入站附件

支持的消息类型：

- `text`
- `image`
- `file`
- `mixed`

附件下载后会落到当前会话的 `chatfile/` 目录。

### 10.2 出站文件回传

回传流程：

1. 由 `send_file.py` 或 `/api/send-file` 提交请求
2. Bridge 校验 Bot、Session、文件路径和大小
3. 文件进入上传队列
4. Bridge 通过 WeCom WebSocket 上传文件分片
5. 上传完成后把文件作为 `msgtype=file` 发回对应聊天

### 10.3 允许回传的目录

默认只允许：

- 当前会话的 `chatfile/`

额外允许：

- `FILE_SEND_ROOTS` 中列出的目录

不自动允许：

- `workDir`
- `/tmp`
- 任意其他普通项目目录

推荐做法：

- 生成给用户看的最终文件时，直接写到 `CHATFILE_DIR` 或 `EXPORT_DIR`

## 11. 调度系统说明

### 11.1 当前语义

当前版本统一用 cron definition 管理调度：

- 一次性任务：转成 `cron + maxRuns=1 + autoDeleteOnDone=true`
- 周期任务：直接使用 cron definition

### 11.2 cron 规则

当前仅支持 5 段 cron：

```text
分 时 日 月 周
```

例如：

- `0 9 * * *`
- `*/10 * * * *`
- `30 18 * * 1-5`

当前实现不支持：

- 6 段带秒 cron
- `intervalSeconds`

### 11.3 触发与并发策略

`misfirePolicy`：

- `fire_once_now`
  说明：错过触发后，下一轮尽快补一次
- `skip_missed`
  说明：错过的触发直接跳过

`concurrencyPolicy`：

- `skip_if_running`
  说明：上一次还没结束时，新的这次跳过
- `enqueue`
  说明：主 bridge 支持；`workspace_bridge` 当前不支持

## 13. 常见问题排查

### 13.1 Bridge 启动了，但真正执行会话时报 Codex 未登录

现象：

- Bridge 进程正常
- 一旦用户发消息，`codex` 子进程失败

排查：

```bash
which codex
codex login status
echo "${CODEX_HOME:-$HOME/.codex}"
```

原因：

- Bridge 以当前 Linux 用户身份拉起 `codex`
- 谁启动 Bridge，谁就必须先登录 Codex

### 13.2 文件发送失败：`filePath is outside allowed roots`

原因：

- 目标文件不在 `chatfile/` 或 `FILE_SEND_ROOTS` 下

处理：

- 把最终文件写到 `CHATFILE_DIR`
- 或配置 `FILE_SEND_ROOTS`

### 13.3 `workspace_bridge.service` 文件发送失败：`request body must be valid JSON`

原因：

- `workspace_bridge.service` 的 `/api/send-file` 收到的请求体不是合法 JSON

处理：

- 确认 `Content-Type: application/json`
- 确认请求体本身是合法 JSON，而不是被 shell 转义破坏

### 13.4 `workspace_bridge.service` 文件发送失败：`bot not connected` / 传输层 `502`

原因：

- `workspace_bridge.service` 对应的 Bot 当前未连上 WeCom WebSocket
- 或即时上传/发送阶段出现底层连接异常、网关类错误

处理：

- 先查看根路径 `/` 返回的 `wecomConnected` / `wecomStatus`
- 如果 `wecomStatus` 正常，但服务内部任务仍异常，再看 `runtimeStatus` / `runtimeLastError`
- 确认 Bot 已成功订阅并保持连接
- 对瞬时网络故障按需重试
### 13.5 定时任务或文件发送报 `session not found`

原因：

- 只传了过期 `sessionId`
- 当前 Bot 下没有匹配的 Session

建议：

- 优先保存并使用 `chatKey`
- `sessionId` 只作为补充兜底

### 13.6 同一个 `chatKey` 匹配多个 Bot

错误通常类似：

```text
chatKey matches multiple bots; provide botName or sessionId
```

处理：

- 在 `/api/send-file` 或调度请求里补 `botName`
- 或改用明确的 `sessionId`

### 13.5 Podman / 容器环境里企业微信连不上

日志示例：

```text
[default] WeCom error: Cannot connect to host openws.work.weixin.qq.com:443 ssl:default [Temporary failure in name resolution]
[default] WeCom disconnected
```

这通常是 DNS 解析失败，不是 `botId` / `secret` 错误。

先查容器内 DNS：

```bash
cat /etc/resolv.conf
getent hosts openws.work.weixin.qq.com
```

临时修复示例：

```bash
printf 'nameserver 223.5.5.5\nnameserver 1.1.1.1\noptions timeout:2 attempts:2\n' > /etc/resolv.conf
getent hosts openws.work.weixin.qq.com
```

如果是 Podman，建议修容器宿主或 `podman machine` 的 DNS 配置，而不是只改运行中容器。

## 14. 安全与部署建议

- 优先监听 `127.0.0.1`，通过反向代理或 SSH 隧道暴露 API。
- 对非本机调用场景，至少开启 `BRIDGE_BASIC_AUTH` 或 `BRIDGE_TOKEN`。
- `CODEX_EXEC_MODE=host` 只适合可信内网环境。
- Secret 一律放 `secretFile`，不要再使用明文 secret。
- 如果服务会重启或容器会重建，使用：
  - `WECOM_BOT_SECRET_FILE`
  - `WECOM_BOOTSTRAP_BOTS_JSON_FILE`
  - `BOTS_FILE` 挂载持久卷
- 如果要让当前用户全局可见，把技能目录放到：
  `~/.codex/skills/<skill-name>/SKILL.md`
- 如果只想让当前工作区可见，把技能目录放到：
  `PROJECT_DIR/.codex/skills/<skill-name>/SKILL.md`

### 14.1 Codex 实际收到的消息示例

Bridge 不会把企业微信原始 JSON 直接发给 `codex`。  
实际传给 `codex` 的是一段拼好的文本 prompt，结构大致如下：

```text

botId: bot-1
chatKey: group-user:room-1:alice
chatType: group
userId: alice
roomId: room-1
sessionId: session-1234567890
workspaceId: user:alice:src_abcd1234
```

如果走旧的 `bridge.py` 路径，BridgeContext 里还会多出：

- `localSendFileCommand`
- `localScheduleMessageCommand`
- `allowedFileSendRoots`

如果当前消息带附件，还会在 `User request:` 之前追加：

```text
Attachment 1: image /path/to/image.png
Attachment note 1: image downloaded from WeCom.
There are 1 attachments still downloading. They may not be available for this answer.
```

因此在调试“为什么 Codex 没命中某个目录 / skill / 文件回传规则”时，优先看这段 BridgeContext 是否符合预期。

## 15. 推荐操作清单

新机器首次部署，建议按这个顺序：

1. 安装依赖：`pip install -r requirements.txt`
2. 验证 `codex` 安装和登录：`which codex && codex login status`
3. 准备 `.env` 和 `secretFile`
4. 如需多 Bot，准备 `WECOM_BOOTSTRAP_BOTS_JSON_FILE`
5. 启动：`sh ./start.sh`
6. 验证：`sh ./check_bridge_health.sh`
7. 如需改核心逻辑，提交前跑：`sh ./smoke_bridge.sh`
8. 回归时跑：`sh ./test.sh`
